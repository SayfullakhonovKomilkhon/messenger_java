# Руководство по ботам мессенджера (для разработчика / другого ИИ)

Документ описывает **роль бота в системе**, **чего сервер не делает сам** и **пошаговую настройку** внешнего сервиса, чтобы бот отвечал пользователям.

Базовый URL API в примерах: `https://<host>/api/v1` — подставьте свой хост (например `http://localhost:3000` для локального backend).

---

## 1. Что такое «бот» в этом проекте

- Бот — это **отдельный пользователь** в БД (`user_id`), привязанный к **владельцу** (`owner_id`).
- У бота есть: имя, опционально `username`, аватар, описание, секретный **токен**, опциональный **URL вебхука**, флаг **активен** (`is_active`).
- В клиенте бот отображается как собеседник в **обычном личном чате 1:1** (как диалог с человеком, но второй участник — учётная запись бота).

**Важно:** мессенджер **не содержит** встроенного ИИ, шаблонов ответов и «эхо». Пока внешний код не вызовет API отправки сообщения, **бот в чате молчит** — это нормальное поведение.

---

## 2. Две части интеграции (обязательно понять обе)

| Часть | Назначение |
|--------|------------|
| **Вебхук (исходящий от сервера мессенджера)** | Когда **человек пишет в чат, где участником является бот**, backend **асинхронно** шлёт **HTTP POST** на сохранённый `webhook_url` с JSON о сообщении. Это **уведомление** для вашего сервиса: «пришло входящее». |
| **Bot API (входящие запросы к мессенджеру)** | Ваш сервис по **токену бота** вызывает REST-методы: отправить сообщение в чат, получить список диалогов, историю, начать диалог с пользователем, задать вебхук. Заголовок: `Authorization: Bot <token>`. |

Связка: **вебхук → ваша логика → `sendMessage` в Bot API → пользователь видит ответ в приложении**.

### 2.1. Не как в Telegram

В Bot API этого мессенджера **нет** inline-кнопок, постоянных клавиатур и сценария «поделиться контактом одной кнопкой». В `sendMessage` передаются в основном **текст** и при необходимости **файл** (`fileUrl`). Телефон и любые данные пользователь обычно присылает **текстом**; разбор — в вашем коде вебхука (пример: репозиторий `tg-bot-testing`).

---

## 3. Когда срабатывает вебхук

Обработчик `BotEventListener` реагирует на событие отправки сообщения в чат:

- Перебираются **участники чата**, **кроме отправителя**.
- Если участник — **бот**, у него `is_active == true`, и задан **непустой** `webhook_url`, вызывается `BotWebhookDispatcher.dispatch`.

**Следствия:**

- На **свои же** исходящие сообщения (когда бот что-то отправил через API) вебхук **на самого бота** не шлётся таким же образом (отправитель пропускается в цикле).
- Если `webhook_url` **не задан** — POST **никуда не уходит**, ответа в чате **не появится**, пока вы не вызовете `sendMessage` сами другим способом.
- Если URL недоступен с сервера мессенджера (firewall, только localhost без туннеля) — в логах будет ошибка доставки, в чате ответа не будет.

---

## 4. Формат тела POST на вебхук

`Content-Type: application/json`. Структура (логически):

```json
{
  "update_type": "message",
  "bot_id": "<UUID бота в таблице bots>",
  "message": {
    "id": "<id сообщения или пустая строка>",
    "conversationId": "<UUID чата>",
    "senderId": "<UUID пользователя-отправителя>",
    "text": "<текст или пустая строка>",
    "fileUrl": "<url файла или пустая строка>",
    "mimeType": "<mime или пустая строка>",
    "createdAt": "<ISO-строка времени или пустая строка>"
  }
}
```

Ваш обработчик должен:

1. Распарсить JSON.
2. Взять `message.conversationId`, `message.text`, `message.senderId` и т.д.
3. Решить, что ответить.
4. Вызвать **`POST /api/v1/bot/sendMessage`** с телом, где есть `conversationId` и хотя бы одно из: `text`, `fileUrl` (см. раздел 6).

## 5. API владельца бота (JWT пользователя)

Используется обычная авторизация пользователя (`Authorization: Bearer <access_token>`).

| Метод | Описание |
|--------|-----------|
| `POST /api/v1/bots` | Создать бота (тело: имя, опционально username, description, avatarUrl). В ответе — `token`, `id`, `userId` и др. |
| `GET /api/v1/bots` | Список ботов владельца |
| `GET /api/v1/bots/{id}` | Один бот |
| `PATCH /api/v1/bots/{id}` | Обновление полей, в т.ч. **`webhookUrl`** |
| `POST /api/v1/bots/{id}/regenerate-token` | Новый токен |
| `DELETE /api/v1/bots/{id}` | Удаление бота |

Мобильное приложение может настраивать вебхук через **редактирование бота** (это уходит в `PATCH` и **сохраняется в БД**).

## 6. Bot API (только токен бота, без JWT пользователя)

Заголовок обязателен:

```http
Authorization: Bot <секретный_токен_из_ответа_create_или_из_приложения>
```

| Метод | Назначение |
|--------|------------|
| `GET /api/v1/bot/me` | Проверка токена; в ответе `id`, `user_id`, `name`, `username`, `is_active` |
| `POST /api/v1/bot/sendMessage` | Отправить сообщение от имени бота. Тело JSON: `conversationId` (UUID), опционально `text`, `fileUrl`, `mimeType`. Нужен **либо текст, либо fileUrl** (или оба). |
| `GET /api/v1/bot/getConversations` | Диалоги бота |
| `GET /api/v1/bot/getMessages?conversationId=...&before=...&limit=30` | История |
| `POST /api/v1/bot/startConversation?userId=<UUID>` | Создать/получить личный чат с пользователем |
| `POST /api/v1/bot/setWebhook` | Тело: `{"url":"https://..."}`. Сохраняет вебхук в БД (пустая строка можно трактовать как очистку — уточняйте по реализации `updateWebhookUrl` на backend). |

Если `is_active == false`, операции отправки/чтения через фасад сообщений для бота **запрещены** (ошибка вроде «бот выключен»).

## 7. Пошаговая настройка (чеклист)

1. **Создать бота** через `POST /api/v1/bots` (или UI приложения). Сохранить **`token`** и при необходимости **`user_id`** бота.
2. **Пользователь открывает чат с ботом** — поиск по username / имя, первое сообщение (как с любым пользователем). Либо бот вызывает `startConversation` с `userId` человека.
3. **Поднять HTTPS (или доступный с backend URL) endpoint** для приёма POST вебхука. Для локальной разработки часто используют **ngrok**, **cloudflared** или сервис вроде **webhook.site** только для отладки приёма (для полного цикла ответа webhook.site нужно комбинировать с ручным/прокси вызовом `sendMessage`, либо свой сервер).
4. **Указать вебхук:**
   - через **`PATCH /api/v1/bots/{id}`** с полем `webhookUrl`, или  
   - через **`POST /api/v1/bot/setWebhook`** с заголовком `Bot <token>`.
5. **Реализовать обработчик вебхука:** прочитать JSON → вызвать **`sendMessage`** с тем же `conversationId`, что пришёл в `message`.
6. **Проверка без вебхука:** один раз вызвать **`sendMessage`** вручную (curl/Postman), подставив `conversationId` из приложения или из логов — если сообщение от бота появилось, токен и чат в порядке, не хватает только шага 4–5.

---

## 8. Минимальные примеры curl

Проверка токена:

```bash
curl -s -H "Authorization: Bot YOUR_BOT_TOKEN" "https://<host>/api/v1/bot/me"
```

Отправка ответа в существующий чат:

```bash
curl -s -X POST "https://<host>/api/v1/bot/sendMessage" \
  -H "Authorization: Bot YOUR_BOT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"conversationId":"CONVERSATION-UUID","text":"Здравствуйте!"}'
```

## 9. Типичные причины «бот не отвечает»

1. **Нет вебхука или URL не сохранён** — события до вашего кода не доходят.
2. **Вебхук не доступен с машины backend** (localhost без туннеля, закрытый порт).
3. **Обработчик вебхука не вызывает `sendMessage`** — сервер мессенджера сам ответ не генерирует.
4. **Бот выключен** (`is_active: false`).
5. **Неверный или просроченный токен** (после `regenerate-token` старый не работает).
6. **В `sendMessage` нет ни текста, ни `fileUrl`** — API вернёт ошибку.

## 11. Краткая формулировка для другого ИИ (one-shot)

> Бот — отдельный пользователь чата. Автоответов в приложении нет. При входящем сообщении в чат с ботом backend POSTит JSON на `webhook_url` бота (`update_type: message`, поле `message` с `conversationId`, `senderId`, `text`, …). Чтобы пользователь увидел ответ, ваш сервис должен вызвать `POST /api/v1/bot/sendMessage` с заголовком `Authorization: Bot <token>` и телом `{ "conversationId": "...", "text": "..." }`. Вебхук настраивается через `PATCH /api/v1/bots/{id}` (`webhookUrl`) или `POST /api/v1/bot/setWebhook`. Бот должен быть активен.
